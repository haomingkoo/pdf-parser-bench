Under the Hood

How parsers actually detect words

Three completely different mechanisms live under the umbrella term "PDF parser." Which mechanism a tool uses determines what it can and cannot read — understanding the internals explains every entry in the benchmark table above.

Lecture — tracing a real PDF file in memory
🎓
Professor
Let me walk through what happens when pypdf opens a digital complaint form. The binary file opens with %PDF-1.4. Inside are numbered objects — think of them as rows in a database. Object 12 is the Page. Object 14 is the Font. Object 15 is the Contents stream — a list of drawing operators like BT (begin text block), Tf (select font), Td (move cursor to position), Tj (draw a text string at the cursor).
🤔
Student
Wait — so text in a PDF is stored as drawing instructions, like a painter following a recipe? Not as actual words in a file?
🎓
Professor
Exactly. And here's the trap: those drawn strings use the font's own character encoding — not plain Unicode. The ToUnicode table inside the font definition converts raw glyph IDs to readable characters. If that table is missing (common in older PDFs), the parser returns garbled characters — even though the PDF looks perfect in Adobe Reader.
🤔
Student
That explains why pypdf shows CER = 0.49 even on a perfectly clean digital form. But its Field F1 is 1.0 — perfect. How?
🎓
Professor
AcroForm field values live in a completely separate location: the /V entry of each widget annotation object — not inside the content stream at all. pypdf has a dedicated path that reads these /V entries directly, bypassing the drawing commands entirely. The high CER is a measurement artefact — the ground truth interleaves labels with values in a specific order that doesn't match the content stream's drawing sequence.
⚡
Expert insight
This is the most common trap in PDF benchmarks: never use CER or WER as the primary metric for AcroForms. CER measures whether two text strings match character-for-character in the same order — but AcroForms are structured data, not prose. The correct metric is Field F1: did you extract the right value for each named field? pypdf scores 1.0 on Field F1, meaning it extracted every form field correctly — which is what actually matters for a complaint processing pipeline.

How Tesseract segments words

  1. Binarize — Otsu global threshold converts grey pixels to pure black/white. Sauvola[8] is better for mixed handwrite/print (adapts per tile).
  2. Find baselines — horizontal projection profiles detect row gaps. Each ink-dense row is a text line.
  3. Find words — vertical projection profiles within each line detect white gaps. Each connected blob is a word candidate.
  4. Classify characters — LSTM (Tesseract 4+) reads each word image left-to-right and predicts character sequences with a CTC loss decoder.
  5. Language model rescoring — a word-n-gram model corrects low-confidence character predictions (e.g. rn → m).

PaddleOCR additions

PaddleOCR replaces Tesseract's heuristic segmentation with a detection model (DB++ — a differentiable binarization network) that directly predicts word bounding boxes as a segmentation map. This handles rotated, curved, or irregularly spaced text that trips up projection profiles.

PP-StructureV2 → table HTML

PP-StructureV2 runs a separate layout model that classifies regions as text / title / table / figure, then applies a table-specific parser to reconstruct row/col structure as HTML.

Why 300 DPI matters: a character that is 3mm tall on paper = 35 px at 300 DPI vs 24 px at 200 DPI. The LSTM window is fixed — at 200 DPI, ascenders and descenders merge into adjacent rows. Below 200 DPI accuracy drops off a cliff (~56% CER on complaint forms).
Lecture — pixel density, binarization, and why preprocessing can hurt
🎓
Professor
Let me explain exactly what the Tesseract LSTM sees. It receives a word-height image crop — typically 48 pixels tall in its default mode. Inside that crop, it reads left to right, predicting character probabilities at each horizontal position. If two characters overlap vertically because the image resolution is too low, the LSTM gets confused — it sees one ambiguous blob where two characters should be.
🤔
Student
Can't we just scan at 600 DPI to give the model more pixels and better accuracy?
🎓
Professor
Higher DPI helps up to a point — but the bottleneck often shifts to binarization. The first real step in any OCR pipeline is converting the image to pure black and white. Otsu's algorithm finds a single global brightness threshold for the entire page. Now, if your complaint form has a coffee stain in one corner and clean white paper elsewhere, that global threshold is simultaneously too high in one area and too low in another.
🤔
Student
Is that what Sauvola binarization fixes? Our preprocessing code uses it...
🎓
Professor
Exactly — Sauvola divides the page into overlapping tiles and computes a local threshold for each one, adapting to local contrast. For fax-quality or coffee-stained scans it is dramatically better. But — and this is what our benchmark showed on FUNSD — on an already-clean 300 DPI scan, Sauvola's local window over-sharpens thin pen strokes. CER went from 0.484 → 0.560. Preprocessing hurt.
⚡
Expert insight
The practical rule: gate preprocessing on measured image quality, not a global flag. In the code, check_dpi() returns the scan resolution. If DPI < 200, or skew angle > 1°, or contrast ratio < 50 — apply the full pipeline (deskew → denoise → CLAHE → Sauvola). Otherwise, feed the raw image to Tesseract. Running the full pipeline on adequate-quality inputs makes it 4× slower with no accuracy gain — a trade-off that is never worth making.

Lecture — why ML layout is a completely different category
🎓
Professor
Docling's core insight is to treat a document page as a 2D photograph and apply object detection — the same technology your phone uses to identify faces. Instead of 'face' or 'car', the output classes are 'text block', 'table', 'heading', 'figure', 'caption', 'footnote'.
🤔
Student
Can't you detect tables more simply, by just looking for horizontal and vertical lines on the page?
🎓
Professor
That works for perfectly typeset PDFs with visible borders — maybe 60% of real-world documents. Real complaint forms often have borderless tables with rows separated only by spacing. Merged cells. Headers spanning three columns. Scanned forms where the ink lines are smudged or missing. Rule-based detection fails on all of these. TableFormer predicts the structure as a sequence of HTML tokens — it has learned every possible table layout from 500,000 training examples.
🤔
Student
If Docling is so much better, why not always use it for everything?
🎓
Professor
Cost and speed. Docling runs a 86-million-parameter vision model on every page. On a CPU, that is roughly 0.5–2 seconds per page. Compare that to pypdf at 415 pages per second for a digital AcroForm. For a pipeline processing 10,000 forms a day, spending 1.2 seconds per page means >3 hours of compute — versus seconds with a text extractor. The right tool depends on the document type.
⚡
Expert insight
Docling's DiT backbone (DeiT-B architecture, 86M parameters) was pre-trained on DocBank (500K document images) and PubLayNet (360K annotated pages). This breadth is why it generalises to complaint form layouts it has never seen. TableFormer adds another ~50M parameters specifically for table structure prediction. The total pipeline — layout detection + table reconstruction + OCR fallback — costs approximately 1.2 seconds per page on Apple Silicon CPU.
When to choose Docling: any document that has complex tables, multi-column layouts, scanned pages, or mixed content where reading order matters. For simple digital AcroForms, Category 1 (text extraction) is 800× faster and equally accurate.

Lecture — why TrOCR never needs to find character boundaries
🎓
Professor
Traditional OCR has a fundamental chicken-and-egg problem: to read handwriting you need to segment it into individual characters, but to know where the boundaries are, you already need to know what the characters look like. Tesseract and PaddleOCR work around this using projection profiles — looking for white vertical gaps between characters. That approach fails completely on cursive script where letters are physically connected with no white gap.
🤔
Student
So how does TrOCR solve this? Where does the letter boundary detection happen?
🎓
Professor
It doesn't happen — that's the elegant part. TrOCR takes the entire field image as input and asks a language decoder: "given all these visual patches, what is the most likely Unicode character sequence?" At each decoding step, cross-attention focuses on whichever patches contain the ink relevant to the current character. For the letter 'J', the decoder attends to the top-left patches where the vertical descending stroke lives. For 'o', it attends to the curved-loop patches. Segmentation is implicit in the attention weights — never explicitly computed.
🤔
Student
So if someone writes 'Jones' in connected cursive with no white gaps at all, TrOCR just reads the whole word and the model figures out the boundaries from context?
🎓
Professor
Precisely — and that is exactly why CER is 2.9% on clean IAM handwriting. Traditional character-segmentation OCR breaks badly on connected script. TrOCR handles it naturally. The limitation is training data: the model learned from curated, single-writer samples. Real complaint forms have degraded ink, mixed print and handwriting, and variable letter forms from hundreds of different writers. CER degrades to 30–60% on these — which is why we run TrOCR per-field, with a coordinate-based crop, not whole-page.
⚡
Expert insight
TrOCR is built on two pre-trained models fused together: BEiT (a 86M-parameter vision transformer trained on masked image patches) acts as the visual encoder, and RoBERTa (a 125M-parameter language model) acts as the text decoder. The cross-attention bridge lets the language model's knowledge of plausible character sequences resolve visually ambiguous ink strokes — "rn" that looks identical to "m" in handwriting gets pushed toward whichever makes a real word in context. This is the advantage of an end-to-end trained encoder-decoder over a purely visual classifier.
The bottom line: Categories 1–4 are not interchangeable. A digital complaint form submitted as AcroForm → Category 1 (text extraction) wins. A faxed, crooked, coffee-stained form → Category 2 + preprocessing. A form with multi-column tables and headers → Category 3. A form with handwritten notes in the margins → Category 4. Production pipelines route to the right category per page, not per document.

Running the full pipeline (deskew → denoise → CLAHE → Sauvola binarization) on FUNSD 300 DPI scans increased CER from 0.484 → 0.560 (+0.076 absolute, +15.7% relative). Sauvola binarization (window=25) over-thresholds fine strokes on adequate-quality 300 DPI images, destroying information that Tesseract could read from the raw scan.

Rule of thumb: Gate preprocessing on measured document quality — not a global boolean flag. Use check_dpi() + measure skew angle before deciding to preprocess. Reserve the full pipeline for fax-quality inputs (≤200 DPI, skew >1°, contrast ratio <50).

How much does routing help? — Single parser vs hybrid pipeline

Routing each document type to the best-fit tool instead of running one parser on everything is the single highest-leverage improvement. These estimates combine locally measured results (✓) with published benchmarks — your numbers will vary.

What routing buys you, concretely:
  • Digital AcroForms → pypdf: 100% Field F1 at 415 p/s. Any OCR tool here wastes compute and accuracy.
  • Scanned 300 DPI → Tesseract (raw): ~52% word accuracy. Docling (projected): ~78–85%. +26–33 pp from the right tool alone.
  • Handwritten fields → TrOCR per-field: ~70% on real forms vs ~20–35% with character-segmentation OCR.
  • Low-confidence → HITL queue: catches the 5–10% of edge cases that degrade any automated pipeline.
The ceiling is not 100%:

Even the best hybrid pipeline hits a wall on sub-200 DPI fax scans, heavy coffee-stained or torn forms, and extreme cursive handwriting from a single writer the models have never seen. The practical ceiling for fully automated extraction on real complaint form corpora is around 90–94% Field F1. The remaining 6–10% should go to Human-in-the-Loop review — cheaper than fine-tuning and more reliable than hoping the model generalises.

✓ Must-do

  • Preprocessing pipeline before any OCR
  • Empirical confidence calibration (not vendor defaults)
  • Field-level metrics, not form-level
  • Pydantic v2 cross-field validation
  • Immutable audit log (CFPB requirement)
  • Presidio PII redaction before storage
  • DPA in place before any cloud API

✗ Anti-patterns

  • One parser for all document types
  • LLMs as primary OCR for numeric fields
  • Trusting vendor confidence as calibrated probability
  • Unstructured for real-time intake (51–140s/form)
  • LlamaParse without enterprise DPA on PII data
  • Benchmarking on vendor sample documents
  • Evaluating form-level accuracy (hides field failures)

⚠ Watch out for

  • Fax-quality scans (<200 DPI) — reject or upsample
  • Few-shot prompt contamination → hallucination
  • VLMs hallucinating chart/signature content
  • PyMuPDF AGPL if redistributing commercially
  • Marker/Surya GPL blocking proprietary products
  • Active learning without OOD validation
  • Static thresholds without recalibration over time

Practical starting point for most teams: Start with Docling locally if you need one general parser for scanned and layout-heavy documents. Add TrOCR fine-tuned on real handwritten complaint fields if handwriting matters. For digital AcroForms specifically, the current local measurements favor pypdf.get_fields() for field extraction speed and accuracy on the synthetic corpus used here. Keep HITL for low-confidence cases rather than assuming one parser will cover every subtype.
Before you start: Annotate 200 real forms (ground truth JSON), run the ablation in sandbox/evaluate.py, measure your actual CER/WER/FER, and calibrate your confidence thresholds empirically.
