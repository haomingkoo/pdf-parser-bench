Under the Hood

How parsers actually detect words

Three completely different mechanisms live under the umbrella term "PDF parser." Which mechanism a tool uses determines what it can and cannot read — understanding the internals explains every entry in the benchmark table above.

Category 1 — Text extraction (pypdf · PyMuPDF · pdfplumber)

No image processing. No machine learning. Pure parsing of PDF's internal byte structure.

Inside a PDF content stream

BT /Helvetica 12 Tf % set font + size 72 680 Td % move cursor to (72, 680) (Complaint Date:) Tj % draw string 180 0 Td % advance right (01/15/2024) Tj % draw value ET

Every character's position is encoded as PDF operators. Tj draws a string. Td moves the cursor. Tf selects a font.

Glyph → Unicode mapping

% ToUnicode CMap (inside PDF font object) /CIDInit /ProcSet findresource begin beginbfchar <0041> <0041> % glyph 0x41 → "A" <0042> <0042> % glyph 0x42 → "B" <F001> <0023> % custom glyph → "#" endbfchar

When a font embeds a ToUnicode CMap, parsers can map glyph codes to characters. When it's missing (common in scanned or form-flattened PDFs), parsers produce mojibake — or nothing.

→

Why this fails on complaint forms: scanned PDFs have no content stream — the page is a raster image object. Text extractors return empty string. Also, AcroForm field values live in /V dictionary entries, not content streams — most extractors need a separate code path for forms.

How Tesseract segments words

  1. Binarize — Otsu global threshold converts grey pixels to pure black/white. Sauvola is better for mixed handwrite/print (adapts per tile).
  2. Find baselines — horizontal projection profiles detect row gaps. Each ink-dense row is a text line.
  3. Find words — vertical projection profiles within each line detect white gaps. Each connected blob is a word candidate.
  4. Classify characters — LSTM (Tesseract 4+) reads each word image left-to-right and predicts character sequences with a CTC loss decoder.
  5. Language model rescoring — a word-n-gram model corrects low-confidence character predictions (e.g. rn → m).

PaddleOCR additions

PaddleOCR replaces Tesseract's heuristic segmentation with a detection model (DB++ — a differentiable binarization network) that directly predicts word bounding boxes as a segmentation map. This handles rotated, curved, or irregularly spaced text that trips up projection profiles.

DB++ detection → word bbox polygons
→
CRNN recognition → character sequence
→
PP-StructureV2 → table HTML

PP-StructureV2 runs a separate layout model that classifies regions as text / title / table / figure, then applies a table-specific parser to reconstruct row/col structure as HTML.

Why 300 DPI matters: a character that is 3mm tall on paper = 35 px at 300 DPI vs 24 px at 200 DPI. The LSTM window is fixed — at 200 DPI, ascenders and descenders merge into adjacent rows. Below 200 DPI accuracy drops off a cliff (~56% CER on complaint forms).
Category 3 — ML layout parsing (Docling · TableFormer)

Treats a document page as a 2-D object to detect with vision transformers, not as a byte stream or a bag of characters.

Layout analysis with DiT / DocBankNet

  1. Patch embedding — the page image is cut into 16×16 pixel patches (ViT-style). Each patch becomes a token in a transformer sequence.
  2. Self-attention — the transformer learns which patches belong together (e.g., a table header patch attends to its column patches across many rows).
  3. Region classification — a detection head predicts bounding boxes and labels: text block, heading, table, figure, caption, footnote.
  4. Reading order — a separate model predicts the correct top-to-bottom, column-aware reading sequence across detected regions.

TableFormer: how tables are reconstructed

Table structure is modelled as a sequence prediction problem. Given a cropped table image, the model predicts an HTML token sequence (<tr><td>...</td></tr>) using an encoder-decoder transformer trained on PubTabNet (500K annotated tables).

Table region crop (from layout step)
→
TableFormer encoder → patch features
→
Autoregressive decoder → HTML tokens
→
Cell text from OCR overlay → final table

This is why Docling handles merged cells and spanning headers — it predicts the HTML structure first, then fills text in, rather than trying to infer structure from text positions.

Category 4 — Handwriting recognition (TrOCR)

Microsoft TrOCR is a Vision-Encoder–Language-Decoder transformer trained entirely on handwritten text images. It never uses sliding windows or character classifiers.

Handwritten field cropfrom form region
→
ViT encoder16×16 px patches → CLS token
→
Cross-attentiondecoder attends to patches
→
Autoregressive decodetoken by token, beam=4
→
BPE token sequence→ text string
What TrOCR gets right: because it reads a field as a whole image (not character by character), connected script and ligatures that break character classifiers are handled naturally. CER ~2.9% on clean IAM handwriting.
What TrOCR gets wrong: real complaint forms have noisy, degraded, variable-size handwriting. CER degrades to 30–60% on real-world inputs. The model was trained on curated single-writer samples. Use it per-field, not whole-page.
The bottom line: Categories 1–4 are not interchangeable. A digital complaint form submitted as AcroForm → Category 1 (text extraction) wins. A faxed, crooked, coffee-stained form → Category 2 + preprocessing. A form with multi-column tables and headers → Category 3. A form with handwritten notes in the margins → Category 4. Production pipelines route to the right category per page, not per document.

Measured finding (2026-04-10): Preprocessing hurts already-good scans.
Running the full pipeline (deskew → denoise → CLAHE → Sauvola binarization) on FUNSD 300 DPI scans increased CER from 0.484 → 0.560 (+7.6%). Sauvola binarization (window=25) over-thresholds fine strokes on adequate-quality 300 DPI images, destroying information that Tesseract could read from the raw scan.

Rule of thumb: Gate preprocessing on measured document quality — not a global boolean flag. Use check_dpi() + measure skew angle before deciding to preprocess. Reserve the full pipeline for fax-quality inputs (≤200 DPI, skew >1°, contrast ratio <50).

Decision Framework

Which parser for which form?

Answers to four questions narrow you to the right tool. Route different form subtypes to different parsers in a hybrid pipeline.

✓ Must-do

  • Preprocessing pipeline before any OCR
  • Empirical confidence calibration (not vendor defaults)
  • Field-level metrics, not form-level
  • Pydantic v2 cross-field validation
  • Immutable audit log (CFPB requirement)
  • Presidio PII redaction before storage
  • DPA in place before any cloud API

✗ Anti-patterns

  • One parser for all document types
  • LLMs as primary OCR for numeric fields
  • Trusting vendor confidence as calibrated probability
  • Unstructured for real-time intake (51–140s/form)
  • LlamaParse without enterprise DPA on PII data
  • Benchmarking on vendor sample documents
  • Evaluating form-level accuracy (hides field failures)

⚠ Watch out for

  • Fax-quality scans (<200 DPI) — reject or upsample
  • Few-shot prompt contamination → hallucination
  • VLMs hallucinating chart/signature content
  • PyMuPDF AGPL if redistributing commercially
  • Marker/Surya GPL blocking proprietary products
  • Active learning without OOD validation
  • Static thresholds without recalibration over time

Cost at scale — cloud vs self-hosted

Monthly Cost Estimate — Structured Extraction (1 form ≈ 3 pages)

Break-even: Docling self-hosted vs Azure/Google ≈ 167K forms/month; vs AWS Textract Forms ≈ 26K forms/month. Source: Published pricing pages (April 2026). Prices change — verify before budgeting.

Top recommendation for most teams: Start with Docling locally (zero API cost, MIT license, best compliance posture). Add TrOCR fine-tuned on 50 of your actual handwritten complaint forms to close the handwriting gap. Route ≥300 DPI digital forms through PyMuPDF AcroForm extraction for speed. HITL at confidence <0.70. The full pipeline costs ~$200–800/month in compute vs $3,000–$30,000/month for equivalent cloud structured extraction volume.
Before you start: Annotate 200 real forms (ground truth JSON), run the ablation in sandbox/evaluate.py, measure your actual CER/WER/FER, and calibrate your confidence thresholds empirically.
